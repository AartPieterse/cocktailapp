# Barkast — one Expo app for Android + web/PWA, local-first, self-hosted

> Design/implementation plan. Status: **approved for build, not yet started.**
> Supersedes the earlier Flutter / static-only draft. This version reflects: a single Expo (React
> Native + react-native-web) codebase that **retires the Angular website**, **no Apple account**
> (iOS reached via web/PWA), local-first with **optional** accounts, and **self-hosting from home**.

## Context

Barkast is a cocktail app whose hero feature is cabinet → makeable: pick the ingredients you have,
see what you can make now (plus "almost", missing 1–2). Today it's an Angular 21 static SPA on
Netlify reading a committed `catalog.json` (152 ingredients / 90 cocktails) generated by
`scripts/build-catalog.mjs` from `iba-cocktails-seed.json`. A NestJS + MongoDB stack exists in
`backend/` as a **local authoring tool** (config, throttler, helmet — but **no auth**). Domain types
live in the `@cocktailapp/shared` workspace package.

### Decisions (locked)

- **One universal client in Expo / React Native (TypeScript).** Targets **Android** (native, Play
  Store) and **web/PWA** (react-native-web + Expo Router static export). The web build is *also the
  iPhone/iPad delivery vehicle* (installable PWA in Safari) and the desktop experience. **The Angular
  site is retired** once the Expo web build reaches parity.
- **No Apple Developer account.** No native iOS app, no App Store, no Mac, no iOS EAS builds. iOS users
  get the full app — including login/sync — through the web/PWA.
- **Google Play** account already held ($25 one-time, paid). Android ships there (and via sideload APK).
- **Local-first, account-optional.** Fully usable offline with **no account** (cabinet, likes, theme
  on-device). Optional email+password sign-in **syncs cabinet + favorites** across all platforms.
- **Backend = extend the existing NestJS + MongoDB** (not a rewrite; Mongo stays). Adds a catalog
  endpoint, JWT auth, per-user sync, and account deletion.
- **Self-hosted from home:** Docker Compose (api + mongo + cloudflared). **TLS terminates at
  Cloudflare's edge** via Cloudflare Tunnel (no inbound ports, hides home IP). Encrypted nightly
  backups. A Cloudflare-managed domain (~€10/yr) gives a stable API hostname to bake into the app.
- **Repo:** new `app/` folder (Expo) as a workspace member, sharing `@cocktailapp/shared`.

### Source-of-truth split
- **Catalog** = shared, read-only: `iba-cocktails-seed.json` → seeded into Mongo → served by
  `GET /api/catalog` **and** snapshotted by `build-catalog.mjs` into the app's offline bundle. One
  content hash stamps all sinks; a CI gate asserts they match (fixes the drift risk).
- **User data** (cabinet, favorites) = per-user, read-write, in Mongo, only for signed-in users.

## Architecture

```
iba-cocktails-seed.json ──build-catalog.mjs (stamps version hash)──> app/assets/catalog.json (offline bundle)
        │  db:seed (same seed, same hash)
        ▼
   MongoDB ◀── NestJS API (self-hosted) ── GET /api/catalog (version+ETag/304)          ─┐
   (catalog +          │                    POST /api/auth/{register,login,refresh}       │ HTTPS
    users)             └── JWT-guarded       GET /api/auth/me                              │ terminated
                                             GET/PUT /api/me/data (cabinet, favorites)     │ at Cloudflare
                                             DELETE /api/me (account+data deletion)        │ edge, via
                                                                                           │ Cloudflare
   ONE Expo codebase (app/):                                                               │ Tunnel
     • Android native  → Google Play / sideload                                            │ (no inbound
     • web/PWA export  → Netlify (iOS via Safari install, + desktop)                       │  ports)
     offline: bundled catalog → local cache → refresh from /api/catalog  ─────────────────┘
     makeable computed on-device (shared computeMakeable); anonymous = local only;
     signed-in = cabinet/favorites sync ⇄ /api/me/data on every platform
```

## Part A — Shared package (`shared/`)

- **Extract makeable into `shared/src/makeable.ts`** as pure `computeMakeable(cocktails, availableIds,
  maxMissing): MakeableResult[]`, ported from `frontend/src/app/services/catalog.service.ts` (exclude
  zero-ingredient cocktails; `missing` = required non-`optional` lines not in the set; keep
  `missingCount <= maxMissing`; sort by count then name). Export from `shared/src/index.ts`. The Expo
  app imports it, guaranteeing web/native parity.
- **`build-catalog.mjs`:** stamp a `version` (SHA-256/12 over ingredients+cocktails, deterministic) and
  emit the offline bundle to `app/assets/catalog.json`. `db:seed` loads the same seed so Mongo's
  `/api/catalog` version matches the bundle.
- **CI gate:** after seed + build from one seed commit, assert `bundle.version === /api/catalog.version`;
  fail the release on mismatch.

## Part B — Backend (NestJS, extend `backend/src/`)

- **CatalogModule** — `GET /api/catalog` → `{ version, counts, ingredients, cocktails }`; set `ETag`,
  honor `If-None-Match` → `304`. Reuses existing `IngredientsService`/`CocktailsService`.
- **AuthModule** — email + password (bcrypt), JWT via `@nestjs/jwt` with **short access-token expiry +
  refresh token + revocation**; `POST /api/auth/register|login|refresh`, `GET /api/auth/me`;
  hand-rolled `JwtAuthGuard`.
- **User + MeModule** — `User { email (unique, CI index), passwordHash }`; per-user doc
  `{ userId, cabinet: string[], favorites: string[], updatedAt }`. JWT-guarded `GET/PUT /api/me/data`
  and **`DELETE /api/me`** (account + data deletion — GDPR + Play requirement).
- **Security wiring (must-fix — see §Security):** `trust proxy` + `CF-Connecting-IP` so rate limits key
  on the real client IP behind the tunnel; a **strict per-account login throttle** (bcrypt DoS guard) on
  top of the global `ThrottlerGuard`; **explicit prod `CORS_ORIGIN` allowlist** (the web origin) — never
  the reflect-any default.
- **AnalyticsModule** — privacy-preserving aggregate analytics. `POST /api/events` ingests **anonymous**
  product events from the app (no user id / device fingerprint; opt-out respected), incrementing
  aggregate counters in an `analytics` collection (daily buckets; per-cocktail / per-ingredient
  tallies) — never a raw identifiable event stream. Operational metrics (request count/latency/errors
  via a Nest interceptor, Mongo size, uptime) are collected in-process. Both are read only by the
  LAN admin dashboard (Part E).
- New deps: `@nestjs/jwt`, `bcrypt` (+ `@types/bcrypt`). Wire in `app.module.ts`; add `JWT_SECRET` (+
  refresh secret) to `backend/.env.example`. Password-reset email (SMTP) deferred.

## Part C — Self-hosting (`deploy/`)

- **Host machine:** a **dedicated, otherwise-unused ASUS VivoBook Pro laptop** (Intel Core / AMD Ryzen
  → AVX2 present, so MongoDB 7 runs; near-idle workload, plenty of headroom). Run **Ubuntu Server /
  Debian + Docker**. Laptop-as-server setup: disable sleep-on-lid-close (`HandleLidSwitch=ignore` in
  `/etc/systemd/logind.conf`); enable ASUS's **battery charge limit (60–80%)** in MyASUS/BIOS so 24/7
  mains power doesn't degrade the battery (which still acts as a built-in UPS for brief outages); add a
  **USB Ethernet adapter** (no built-in RJ45) for a reliable wired link; ensure airflow; keep Docker
  volumes + Mongo data on the internal **NVMe**. Idle draw ~10–20 W (≈ €30–60/yr).
- **`backend/Dockerfile`** — multi-stage (repo-root context): build `@cocktailapp/shared` then
  `backend`; run `node dist/main`.
- **`deploy/docker-compose.yml`** — services:
  - `api` — built image; `.env`: `MONGODB_URI` (with credentials), `JWT_SECRET`, refresh secret,
    `CORS_ORIGIN`.
  - `mongo` — `mongo:7`, **`expose`-only (no host port published)**, **auth enabled**, named volume.
  - `cloudflared` — Cloudflare Tunnel → stable public hostname (token via env). **TLS terminates at
    Cloudflare's edge** (optionally a Cloudflare Origin cert to the box). **No Caddy ACME** — Let's
    Encrypt would need inbound 80/443 the tunnel deliberately avoids. **Only the public user routes
    (catalog/auth/me) are mapped into the tunnel** (outbound-only, so the box can still pull images).
  - `admin-dashboard` — small metrics UI **bound to the LAN only** (e.g. `:8080`), **never mapped into
    the cloudflared ingress**, so it's reachable from your PC at home but invisible to the internet.
    Publish `barkast.local` via avahi/mDNS; protect with basic-auth.
- **Backups** — nightly `mongodump`, **encrypted (age/gpg) before leaving the box**, off-box storage
  with restricted ACLs; **test restores**.
- **Hardening** — separate VLAN, rootless containers, patched images; a stable **Cloudflare-managed
  domain** (~€10/yr) for the hostname baked into the shipped app.
- Seed once on first bring-up (`npm run db:seed` against the container Mongo).

## Part D — Expo app (`app/`)

### Stack
- **Expo** (latest SDK) + **Expo Router** (file-based nav) + TypeScript; **web target enabled**
  (react-native-web, static export). Depends on `@cocktailapp/shared`; `app/` added to root
  `workspaces` with **Metro monorepo config** (`watchFolders` = repo root, `nodeModulesPaths`).
- **State: Zustand + persistence** for `cabinet`, `favorites`, `theme`, `wizardDone` (mirrors the web
  signal services; `barkast.*` keys → MMKV on native / IndexedDB/localStorage on web).
- **TanStack Query** for `/api/catalog` refresh + `/api/me/data` sync (retry/cache); a small `fetch`
  client. **Auth token:** SecureStore on native; on web, in-memory access token + refresh (or an
  httpOnly cookie if the web is co-hosted on the API domain — see note). Login is infrequent (sync
  opt-in only), so token UX is low-traffic.
- **Offline catalog:** bundle `app/assets/catalog.json`; cache refreshed copy (`expo-file-system` /
  Cache API on web); **makeable computed locally** → works with no network on every platform.
- **UI:** NativeWind (Tailwind-for-RN, works on web) mapping the current editorial palette; light/dark;
  `@expo/vector-icons`. App icon / splash / PWA manifest + service worker for installability + offline.

### Local-first sync
- Zustand is the on-device working copy (always). Anonymous = local only, zero network.
- **On sign-in:** pull `/api/me/data`, **merge** (union of sets on first link), write back; thereafter
  server-authoritative with debounced `PUT`.
- **On app open (authed + online):** pull + merge. Offline edits queue and push on reconnect.
- Same code on Android + web ⇒ **the same account shows the same cabinet/favorites everywhere.**

### Website cutover (retiring Angular)
Build the Expo web export to parity → deploy it to the existing Netlify site (replacing the Angular
bundle + `_redirects` SPA rule) → then delete `frontend/`. The old `frontend/public/catalog.json` is
replaced by the Expo bundle asset. Keep the git-triggered Netlify deploy; repoint it at the Expo web
export. (Web co-hosting on the API domain is an option if you prefer httpOnly-cookie auth over CORS.)

## Part E — Analytics & LAN admin dashboard (owner-only, over the LAN)

Public users reach the API over the tunnel; **you** read usage data over the home LAN only.

- **Product analytics (anonymous, aggregate):** the app sends lightweight events (respecting an opt-out)
  to `POST /api/events`; the backend keeps only aggregates — active-device count, most-added
  ingredients, most-makeable & most-favorited cocktails, wizard-completion rate, catalog refreshes.
  No personal identifiers, no per-user tracking, no raw event log.
- **Operational metrics:** request counts / error rates / latency (Nest interceptor), Mongo size,
  uptime — collected in-process (optionally exported to Prometheus/Grafana later; not needed for v1).
- **LAN admin dashboard:** a small web UI the laptop serves **on the LAN only** (e.g.
  `http://barkast.local:8080`, published via avahi/mDNS) showing both sets above. **Not added to the
  cloudflared ingress**, so it is unreachable from the public internet; protected with **basic-auth**.
  Plain HTTP on the LAN is fine here (no PWA/service-worker needs); TLS optional via the DNS-01 cert.
  The split is enforced simply by not mapping that port into the tunnel config.

## Part F — Working on it: dev/prod, deployments & database updates

**Two environments — never develop against prod.**
- **Local dev** (your Windows PC): backend + a local Dockerised Mongo with a *separate DB name* and its
  own `.env`; the Expo app points at `localhost` / your PC's LAN IP. Iterate freely; never touches the
  laptop or real user data.
- **Production**: the laptop stack (Part C).

**Deployments are low-stakes because the app is local-first** — a brief API restart only pauses sync;
every client keeps working from its cache, so short downtime is invisible to users.
- GitHub Actions builds + tests on push and **pushes a version-tagged image (git SHA) to GHCR**.
- The laptop **pulls** it (outbound-only, works behind the tunnel with no inbound ports) via a
  one-command `deploy.sh`: `docker compose pull && docker compose up -d` (`--no-deps api` for API-only).
  Keep the previous tag for instant **rollback**. (Watchtower auto-pull is possible but I'd keep deploys
  manual given DB migrations. Near-zero-downtime via two API replicas is a later option, not needed v1.)

**Database updates — the safe split:**
- **Catalog:** edit `iba-cocktails-seed.json` → `build-catalog.mjs` (new version hash) → `db:seed`
  reseeds **only the catalog collections (ingredients/cocktails)** — it must **never touch
  `users` / `me-data` / `analytics`**. Clients pick up the change via `/api/catalog` version/ETag on
  next refresh; the offline bundle updates on the next app release.
- **User-data / schema:** versioned migrations (`migrate-mongo`) run by `deploy.sh` **after an automatic
  pre-migration `mongodump`**. Mongo is schemaless so additive changes are trivial; destructive ones
  require the backup + a tested rollback. Take an on-demand encrypted dump before any risky change.
- **Config separation:** distinct `.env` per environment (DB name, `JWT_SECRET`, `CORS_ORIGIN`, tunnel
  hostname); dev never shares prod secrets.

## Security & compliance (must-do before storing anyone's PII)
1. **Real client IP behind the tunnel** — `trust proxy` + `CF-Connecting-IP`; add a strict per-account
   login throttle (protects the bcrypt compute-DoS path).
2. **Lock down Mongo** — never publish port 27017; enable auth; credentials in `MONGODB_URI`.
3. **Encrypt nightly backups** (age/gpg) — dumps contain emails + password hashes; test restores.
4. **JWT lifecycle + CORS** — short access expiry + refresh + revocation; explicit prod `CORS_ORIGIN`
   allowlist (never reflect-any).
5. **TLS at Cloudflare's edge**, not Caddy/ACME (which would need the inbound ports the tunnel avoids).
6. **GDPR (NL data controller once anyone signs in)** — `DELETE /api/me` + a published **privacy policy**.
7. **Google Play** — high **content rating for an alcohol app**; a **working demo account** and a
   **backend that's up during review** (a flaky home origin = rejection).
8. **Residential ISP terms** — confirm hosting others' accounts from home is allowed.
9. **Content licensing** — confirm rights for any bundled cocktail images / IBA-derived content.
10. **Analytics privacy** — anonymous + aggregate + opt-out + disclosed in the privacy policy (no
    per-user tracking); the analytics/admin surface is LAN-only.
11. **Admin dashboard access** — basic-auth even on the LAN; the admin port is excluded from the tunnel.
12. **Deploy secrets & isolation** — GHCR pull token + per-environment `.env`; dev never uses prod
    secrets or the prod DB.

## Verification
- **Backend (e2e):** register/login issues JWT; refresh/revoke works; `PUT`→`GET /api/me/data`
  round-trips and is isolated between users; `DELETE /api/me` erases account+data; `/api/catalog`
  returns `version` and `304` on matching `If-None-Match`. **Security:** Mongo unreachable from host;
  rate limit keyed on real IP; CORS rejects non-allowlisted origins; expired token rejected. Shared
  `computeMakeable` unit test mirrors `backend/src/cocktails/cocktails.service.spec.ts`.
- **App — anonymous/offline (core promise), on Android AND web/PWA:** airplane mode, fresh install →
  onboarding → wizard (staples pre-checked) → correct makeable counts → favorite + cabinet persist
  across restart, with **no network and no account**.
- **Web↔mobile parity (the fixed gap):** sign in on Android and on the web/PWA → same cabinet/favorites
  on both; merge correct; sign out retains local.
- **Catalog:** CI hash gate green; change seed → reseed → app pulls new `version`; offline serves cache.
- **Self-host:** `docker compose up` → API reachable over HTTPS via the tunnel hostname; Mongo persists;
  encrypted backup produced and **restore tested**.
- **Analytics/admin:** the LAN dashboard is reachable from your PC (`barkast.local:8080`) showing
  product + operational metrics, but is **NOT reachable via the public tunnel hostname**; the app
  opt-out halts event sending; only aggregates are stored (no identifiers).
- **Deploy/rollback:** a version bump deploys via `deploy.sh` with only a brief sync pause (the app
  stays usable offline); redeploying the previous image tag rolls back; a `db:seed` reseed leaves
  `users` / `me-data` / `analytics` untouched; a user-data migration runs only after an automatic
  pre-migration dump.
- Same cabinet → identical makeable list on Android, web, and (sanity) the backend aggregation.

## Monetization (optional, low-friction — build only if wanted)
- **Affiliate barware/glassware links** — physical goods are exempt from in-app-purchase rules, so a
  tasteful "Gear for this drink" (shaker, jigger, glass) earns commission at **0% store fee**. Note:
  affiliate programs typically pay **0% on alcohol** — target barware, not bottles. Verify NL rates
  (Bol.com / Amazon.nl).
- **One-time "Support Barkast / Pro"** (~€2.99–4.99) unlocking a cosmetic theme pack — surfaced once,
  never nagged. On **web via Stripe (0% store fee, and it serves iPhone users too)**; on **Android via
  Play billing (15% under the Small Business Program)**. RevenueCat (free tier) can manage entitlements.
- **No banner ads** (negligible eCPM in a niche; breaks the offline/no-tracking identity).
- The **web is the cheapest surface** (no gatekeeper, 0% commission) and is your primary paid channel.
- Receiving money brings NL/EU tax/KvK/VAT considerations — check before enabling.
